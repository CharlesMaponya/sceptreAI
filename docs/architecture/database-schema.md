# Database Schema

The schema is designed around one hard rule: `project_id` is the tenant isolation boundary. Human-readable project names may repeat.

## Core Relationships

```text
User
 ├── Project.owner_id
 ├── Project.created_by_id
 ├── ProjectMembership.user_id
 ├── Dataset.created_by_id
 └── ModelRun.created_by_id

Project
 ├── ProjectMembership
 ├── ProjectShareLink
 ├── Dataset
 │    └── DatasetVersion
 │         └── ModelRun
 │              ├── Metric
 │              ├── RunArtifact
 │              └── ModelRegistryEntry
 └── Project-scoped metrics, artifacts, and registry entries
```

## Tables

### `users`

Stores account identity, simple-auth or SSO source, global admin/member role, password hash when applicable, and token rotation state.

Important fields:

- `id`: UUID primary key
- `email`: unique login identity
- `auth_provider`: `simple` or `sso`
- `global_role`: `member` or `admin`
- `token_version`: invalidates issued tokens when incremented

### `refresh_tokens`

Stores hashed refresh tokens grouped by `family_id` so 24-hour silent rotation and family revocation can be implemented without storing plaintext tokens.

### `password_reset_tokens`

Stores hashed reset tokens with expiry and single-use tracking.

### `projects`

Stores project metadata. `name` is not unique by design.

Important fields:

- `id`: UUID isolation key
- `owner_id`: accountable owner
- `created_by_id`: creator audit field
- `object_prefix`: optional stable object-storage prefix, usually derived from project UUID
- `settings`: JSONB project-level overrides

### `project_memberships`

Defines project-level RBAC. The unique constraint on `(project_id, user_id)` prevents duplicate memberships while still allowing project names to repeat globally.

### `project_share_links`

Stores hashed invite tokens, target role, optional permission overrides, expiry, use count, and revocation state.

### `datasets`

Represents the logical dataset inside a project. Versions are stored separately so model runs can bind to an immutable dataset version.

### `dataset_versions`

Stores immutable dataset lineage:

- `project_id` and `dataset_id`
- `version_number`
- source file format and object URI
- content hash, row/column counts
- inferred schema, type information, and quality report JSON
- profile artifact URI for heavier plots or reports

The composite foreign key `(project_id, dataset_id)` ensures a dataset version cannot point at a dataset from another project.

### `model_runs`

Stores one executable pipeline request, including training, validation, explainability, drift, and deployment jobs.

Important fields:

- `project_id`
- `dataset_version_id`
- `task_type`
- `status`
- `target_column`
- `mlflow_run_id` and `zenml_run_id`
- Kubernetes job name/namespace
- CPU, memory, GPU, and estimated core-hour fields
- user-facing failure translation

The composite foreign key `(project_id, dataset_version_id)` prevents a run from using a dataset version from another project.

### `metrics`

Stores run metrics scoped by project and split. Supports scalar metrics in `value` and structured diagnostics in `value_json`.

Examples:

- validation ROC-AUC
- test RMSE
- silhouette score
- drift score
- resource usage summary

### `run_artifacts`

Stores references to object-store artifacts generated by jobs:

- diagnostic plots
- model binaries
- SHAP outputs
- log bundles
- deployment images

### `model_registry_entries`

Stores project-isolated model registry state. Registry entries link back to the winning run and concrete model artifact, include a stage, and carry a `feature_space_hash` for fallback model selection.

## Query Safety Pattern

Every API query for project data should include `project_id` from the authorized workspace context:

```python
select(ModelRun).where(
    ModelRun.project_id == authorized_project_id,
    ModelRun.id == requested_run_id,
)
```

Do not query isolated records by their UUID alone in request handlers.
